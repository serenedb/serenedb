# False positive lock-order-inversion in the serenedb catalog store path.
#
# TSAN reports a cross-thread deadlock between DROP COLUMN's DataTable
# copy-construct (LIdx -> LMeta -> LM) and that same txn's commit (LM -> LIdx),
# triggered by `ALTER ... DROP COLUMN` / catalog Drop/ChangeTable
# (Catalog::DropTableColumn / DropType / ChangeTable). It cannot deadlock:
#   - All real DuckDB storage locks (DuckTransactionManager transaction_lock,
#     DataTable::append_lock, TableIndexList lock) live ONLY in the hidden
#     __sdb_store database. User tables use SereneDBTransactionManager (no-op
#     commit, owns no DataTable), so they take none of these locks.
#   - Every store mutation runs through CatalogStore::ExecuteEntries, which holds
#     absl::MutexLock{&_mutex} (server/catalog/store/store.cpp) around the whole
#     BEGIN..ALTER..COMMIT. Both reported lock-order edges execute SEQUENTIALLY
#     on one thread under _mutex, so the lock instances never form a cross-thread
#     cycle. TSAN's deadlock detector is instance-agnostic and unaware of the
#     outer _mutex, hence the false report.
# Caveat: only valid while __sdb_store is the sole writable DuckDB-native
# transaction manager. Re-check if another native-DuckDB writable txn manager
# is introduced.
deadlock:sdb::catalog::CatalogStore::ExecuteEntries
